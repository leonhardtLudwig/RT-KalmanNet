## 2026-07-08 ‚Äî Option B: Partial-Information experiment (in progress)

**Files**: `main_Robust_KNet.ipynb` (11 new cells, inserted after the
existing Task 4 section)

Added a second, parallel experiment: filters (REKF, RT-KalmanNet,
KalmanNet) now operate with a **mismatched** internal model
(`sys_model_partial`, using the original KalmanNet paper's Table II
"Partial" parameters: `Œ±=1,Œ≤=1,œÜ=0,Œ¥=0`, `a,b,c` unchanged) while data is
still generated by the true "Full" model ‚Äî a genuine, persistent per-step
mismatch, unlike the earlier one-time `m1x_0` offset. Goal: give `c_t` a
scenario with real signal to adapt to (the full-info SNL benchmark has
none ‚Äî see Findings).

New cells cover: defining `f_partial`/`h_partial`, building
`sys_model_partial` (with `.alpha/.beta/.phi/.a/.b/.c` set to match, since
`RobustKalman`'s hard-coded Jacobian reads those directly), a sanity check,
REKF/RT-KalmanNet evaluation under the mismatch (reusing already-generated
train/CV/test data), a `c` sensitivity sweep, and an optional
KalmanNet-under-partial-info cell.

**Two bugs hit and fixed while bringing this up:**
1. Sanity-check cell incorrectly asserted `h_partial != h_full` ‚Äî wrong
   assumption; per the paper's Table II, only `f`'s constants differ
   between Full and Partial, `a/b/c` (which define `h`) are identical by
   design. Fixed the assertion to only check `f`.
2. `sys_model_partial.m` came out `0` (should be `2`) because it was built
   from the notebook's bare `m` variable, which had drifted to `0`
   somewhere in the long-running kernel session (root cause not
   pinned down via static search). This silently broke KalmanNet's
   `NNBuild` (`nn.GRU(..., hidden_size=0)` ‚Üí `ValueError`) but did *not*
   affect REKF/RT-KalmanNet, since `RobustKalman` derives its dimensions
   from `Q`/`R` shapes, not `SysModel.m`/`.n`. Fixed by building
   `sys_model_partial` from `sys_model.m`/`sys_model.n` (proven correct,
   since Cell 35's original KalmanNet run already depends on them)
   instead of the bare variables. **Status: fix applied, not yet
   re-verified** (KalmanNet-under-partial-info cell still pending a
   re-run ‚Äî deferred, ~200-epoch training run).

**Result so far** (REKF, RT-KalmanNet ‚Äî both ran successfully): see
Findings ¬ßOption B for full interpretation. Headline: MSE degrades 55√ó
under the mismatch (0.0137‚Üí0.757, confirming it's real and severe), but
`c`/`c_t` shows **zero** measurable sensitivity to it (0.0% sweep spread,
`c_t` frozen across 200 epochs) ‚Äî likely explained by `c_range=0.2` being
too narrow and/or `fnComputeTheta`'s bisection fallback branch
(`if not (g_low<=0 and g_high>=0): theta_hat = 0.5*high`) silently
discarding `c` when `P` is small. Not yet disambiguated.

## 2026-07-08 ‚Äî Performance & measurement-fairness fixes (round 2)

**Files**: `RobustKalmanPY/robust_kalman.py`, `main_Robust_KNet.ipynb`

Found while running the Phase 1 fix at full scale (`N_T√ónum_of_test_sets =
10,000` sequences):

- `robust_kalman.py` ‚Äî `fnComputeTheta`'s Newton refinement now only runs
  when `self.use_nn=True`; skipped for the plain REKF baseline
  (`use_nn=False`), which never needs a gradient through `Œ∏`. Restores
  REKF's original per-step speed (was paying for unused
  `autograd.grad()` work after the Phase 1 fix made the refinement
  unconditional).
- `robust_kalman.py` ‚Äî `fnREKF` now also tracks the filtered/posterior
  estimate (`self.Xn_out`, uses `y_t`), alongside the existing
  one-step-ahead predicted estimate ‚Äî needed so REKF/RT-KalmanNet can be
  compared fairly against KalmanNet's posterior-based metric (previously
  apples-to-oranges).
- `main_Robust_KNet.ipynb`, Cell 15 ‚Äî `test_data_synt` is now
  pre-allocated (`pd.DataFrame(index=range(n), ..., dtype=float)`) instead
  of grown one row at a time via `.loc[i, ...]` on out-of-bounds indices ‚Äî
  fixes an `O(n¬≤)` pandas anti-pattern that had turned a ~60-90 minute
  evaluation into a 7+ hour one.
- `main_Robust_KNet.ipynb`, Cell 5 ‚Äî `num_of_test_sets` reduced `100‚Üí10`
  for iteration (statistically sufficient given observed per-sequence MSE
  std ‚âà `0.0006`; bump back up for a final reporting run).
- `main_Robust_KNet.ipynb`, Cells 18/27 ‚Äî now also compute and report the
  filtered MSE (via the new `Xn_out`) alongside the existing predicted
  MSE.

## 2026-07-07 ‚Äî `c_t` collapse fix (Phase 1)

**Files**: `KNet/RT_KalmanNet_nn.py`, `RobustKalmanPY/robust_kalman.py`,
`main_Robust_KNet.ipynb`. **Commit**: `c473edc`.

Fixed the root cause of `MSE(RT-KalmanNet) ‚âà MSE(REKF)`: the learned
tolerance `c_t` had collapsed to an almost-constant value
(`mean=0.108997, std=0.001245`) sitting exactly at the network's
architectural default, not a genuinely adaptive signal.

- `RT_KalmanNet_nn.py` ‚Äî added `c_init` constructor param; output head's
  bias now initialized via inverse-sigmoid so the untrained network's
  default output is a principled `c_init` (defaults to REKF's own `c`),
  not an arbitrary range-midpoint.
- `robust_kalman.py` ‚Äî `c_init` threaded through; precomputed
  `_obs_scale`/`_state_scale` (from `R`/`Q` diagonals) used to rescale the
  4 input features before they reach the network (fixes a 100√ó unit
  mismatch between observation- and state-space features); new
  `reset_state()` method called automatically at the top of `fnREKF`
  (`reset=True` default), removing the need for callers to manually reset
  4 attributes before every call; `fnComputeTheta`'s Newton refinement
  made to always run (train/eval value-consistency) ‚Äî later partially
  reverted for REKF speed, see 2026-07-08 above; `self.th`
  (previously allocated, never populated) and a new
  `theta_residual_array` now populated for diagnostics.
- `main_Robust_KNet.ipynb` ‚Äî optimizer switched to param groups excluding
  `fcl_out` from weight decay (the primary fix ‚Äî L2 shrinkage on the
  output head was what pulled `c_t` to its collapse point); gradient
  clipping added; 4 duplicated manual state-reset blocks deleted (now
  automatic); new diagnostic cells added (`c_t` mean/std vs. epoch,
  constant-`c` sensitivity sweep, `Œ∏_t` trace, train/eval solver-coherence
  check).

Full before/after numbers in [FINDINGS.md](FINDINGS.md).
