## Option B — Partial-Information experiment (2026-07-08)

**Setup**: filters operate on a *mismatched* internal model
(`sys_model_partial`, Table-II "Partial" params for `f` only — `h` is
identical to Full by design) while data is generated by the true model.
Reuses already-generated train/CV/test data; only the filter's internal
model differs from the full-info experiment.

**[GOOD]** The mismatch is real, persistent, and severe: MSE degraded
**55×** (`0.0137 → 0.757`, filtered), vs. essentially no degradation from
the earlier one-time `m1x_0` mismatch (`0.0139 → 0.0137`). Confirms the
experimental design succeeded at creating a scenario Task 4's original SNL
benchmark didn't have.

**[BAD / surprising]** `c` — constant *or* learned — has **zero measurable
effect** on this mismatch. Constant-`c` sweep is identical to 6 decimals
across the entire `c ∈ [1e-4, 0.2]` range (`0.756969` every time, 0.0%
spread, vs. ~0.3% in the full-info experiment). `c_t` during 200 epochs of
training moved from `0.00100` to `0.00099` — statistically frozen.

**[OBSERVATION]** Two candidate mechanisms for the above, not yet
disambiguated:
1. `fnComputeTheta`'s bisection has a fallback branch
   (`if not (g_low<=0 and g_high>=0): theta_hat = 0.5*high`) that is
   **`c`-independent**. Since `P` here is small (`Q`-dominated,
   `≈0.001`-scale), `θ_hi` sits close to `1/λmax(P)`, where
   `logdet(I-θP)→-∞` — plausibly making `g_high<0` for the entire tested
   `c` range, silently triggering this fallback every time regardless of
   `c`.
2. `c_range=0.2`'s ceiling (already flagged as ~5-10× too narrow for
   Task 3's LF_DATA, `c≈1`) may simply not reach the regime where `θ`
   would respond, independent of (1).
   Both point at the same next action (widen `c_range`, re-sweep) — see
   [ROADMAP.md](ROADMAP.md).

**[OBSERVATION]** This *revises* an earlier claim (see "Phase 2 structural
analysis" below): REKF isn't necessarily architecturally incapable of
correcting this kind of mismatch. Since `h_partial == h_full` here (only
`f` is wrong, observations are unbiased), a large-enough `θ` (→ fully
trust observations, ignore the biased prediction) should in principle be
able to compensate — this is a case where the ceiling looks more like
"haven't reached the useful `c` range yet" than "fundamentally capped by
linearization." Worth re-testing once `c_range` is widened before
concluding either way.

**[BAD, not yet re-verified]** KalmanNet-under-partial-info hit a bug
(`sys_model_partial.m` was `0`) that's been fixed in code but not
re-confirmed by an actual run (200-epoch training, deferred). Expected to
work once re-run — see CHANGELOG for the fix.

## Round 2 — Performance & fairness fixes (2026-07-08)

**[GOOD]** REKF's per-sequence time dropped `0.457s → 0.320s` after gating
the Newton refinement behind `use_nn`. Side effect: resolved a timing
paradox from the previous run where RT-KalmanNet had measured *faster*
than REKF (architecturally backwards) — now REKF < RT-KalmanNet, as
expected once REKF stopped paying for unused gradient work.

**[GOOD]** Cell 18's evaluation dropped from 7+ hours to fast (single-digit
minutes) after the pandas pre-allocation fix + `num_of_test_sets`
reduction. Root cause was `O(n²)` DataFrame growth, not the REKF math
itself (which was always ~60-90 min worth of real compute for 10,000
sequences).

**[OBSERVATION → ruled out a hypothesis]** Fixing the KalmanNet-vs-REKF
metric asymmetry (predicted vs. filtered) only closed **~1.5%** of the
2.4× MSE gap (`0.0139→0.0137` for REKF/RTK). This *rules out* "the gap is
mostly a measurement artifact" — it's real and structural. See "why the
gap persists" below (from the Phase 2 plan, still holds for the
full-information SNL benchmark specifically).

## Phase 1 — `c_t` collapse fix (2026-07-07)

**[BAD, fixed]** `c_t` had collapsed to `mean=0.108997, std=0.001245` —
the network's architectural default under a zero-initialized bias, not a
learned signal. Three compounding causes: (1) `weight_decay` applied to
the output head pulling it toward 0, (2) an inherently weak (but legitimate
— implicit-Newton) gradient path from the loss to `c_t`, (3) no feature
normalization across a 100× scale gap between observation- and state-space
features. Fixed in all three; see CHANGELOG for the exact edits.

**[GOOD]** Fix validated at three independent scales (smoke-test, first
full run, second full run): `c_t` now consistently settles near `c_init`
(`≈0.001`, matching REKF's own baseline), not the old collapse point.
Train/eval solver coherence is exact (`0.0` difference between grad-enabled
and `no_grad` passes) — confirmed at both full runs.

**[OBSERVATION]** `REKF ≈ RT-KalmanNet` on the full-information SNL
benchmark is **not a remaining bug** — confirmed via the sensitivity sweep
at three scales (n=2,500 up to n=1,000,000 timestep-samples), MSE varies
by only ~0.3% across the *entire* representable `c` range. This matches
Zorzi's own model-perturbations paper (Section VII-B): when the actual
mismatch is small relative to tolerance, robust and plain filters
converge. The pre-fix "RT-KalmanNet beats REKF 100% of the time" result
was an artifact of the collapsed network landing on a constant marginally
better than `c=1e-3`, not genuine adaptivity.

### Why the KalmanNet vs. REKF-family gap persists (full-info SNL benchmark)

This project's SNL benchmark uses the exact same "Full information"
parameters as the original KalmanNet paper's own study. Their own tables
show EKF and KalmanNet close under full information, but KalmanNet pulling
dramatically ahead under partial information (parameter mismatch). Two
reasons this project's gap (`~2.4×`) is larger than pure full-info numbers
would predict:
1. REKF is still EKF-based (local linearization) — `c`/`θ` only rescales
   trust between prediction and observation, it can't correct bias coming
   from linearizing around the wrong point. KalmanNet learns the gain
   directly, unconstrained by this.
2. This benchmark already has a narrow mismatch — filters use
   `m1x_0=zeros`, true generative `m1x_0=ones` — a real but *one-time*
   initial-condition offset, not persistent, so a robustness parameter has
   little ongoing reason to respond to it (contrast with Option B's
   *persistent* mismatch above, which prompted a re-examination of claim
   (1) — see the Option B observation above).
